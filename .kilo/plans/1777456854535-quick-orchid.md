# Video Playback Architecture Refactoring Plan

## Objective
Decouple the video detail page's UI from playback logic by introducing a service-oriented architecture with library-specific wrappers and a unified video player component.

## Background
- `video_detail_page.dart` currently embeds all player initialization, DASH audio setup, heartbeat, and control logic.
- The app supports two player backends: `media_kit` (default) and `fvp`.
- `MediaS` exists as a singleton but is mostly a stub.
- `PlayerLibraryM` enum and app settings infrastructure already exist.

---

## Files to Create

### 1. `lib/feature/video/widget/media_kit_video_v.dart`
- **Purpose**: Lightweight wrapper around `media_kit`'s `Video` widget.
- **Parameters**:
  - `VideoController controller` (required)
- **Implementation**: Stateless widget that returns `Video(controller: controller)`.
- **Encapsulation**: Only imports `media_kit_video`; no business logic.

### 2. `lib/feature/video/widget/fvp_video_v.dart`
- **Purpose**: Lightweight wrapper around `video_player`'s `VideoPlayer` widget (used by fvp).
- **Parameters**:
  - `VideoPlayerController controller` (required)
- **Implementation**: Stateless widget that returns `VideoPlayer(controller)` wrapped in `AspectRatio` if needed.
- **Encapsulation**: Only imports `video_player`; no business logic.
- **Note**: `video_player` is a transitive dependency via `fvp`. We will add it to `pubspec.yaml` explicitly for clarity.

### 3. `lib/feature/video/widget/video_player_v.dart`
- **Purpose**: High-level unified video player component using a `Stack` layout.
- **Structure**:
  - **Bottom layer**: Dynamically switches between `MediaKitVideoV` and `FvpVideoV` based on `MediaS.i.currentLibrary`.
  - **Top layer**: Interaction/HUD overlay (`_VideoHud`) that dispatches all user actions to `MediaS`.
- **Parameters**:
  - `PlayUrlModel? playUrl` (optional, for quality picker)
  - `VoidCallback? onQualityTap`
  - `VoidCallback? onFullscreen`
- **State Management**: Listens to `MediaS` streams (playing, position, duration, buffering) via `StreamBuilder`/`ValueListenableBuilder`. Does NOT manage local playback state.

## Files to Modify

### 4. `lib/service/media_s.dart` (Major refactor)
Expand `MediaS` to own all playback lifecycle management.

#### Internal Player Management
- `_mkPlayer`: `Player?` (media_kit instance)
- `_mkController`: `VideoController?` (media_kit controller)
- `_fvpController`: `VideoPlayerController?` (fvp/video_player instance)

#### Unified State Exposure
Expose reactive state so the UI can listen without touching native APIs:
- `Stream<bool> get playingStream`
- `Stream<Duration> get positionStream`
- `Stream<Duration> get durationStream`
- `Stream<bool> get bufferingStream`
- `PlayerLibraryM get currentLibrary`
- `bool get isInitialized`

#### Playback API (called by UI/HUD)
- `Future<void> initAndLoad(PlayUrlModel playUrl)`
  - Reads `StorageS.getSetting().playerLibrary`
  - Disposes any existing player
  - Initializes the correct backend
  - Loads video URL + DASH audio (backend-specific)
  - Starts playback
- `Future<void> play()`
- `Future<void> pause()`
- `Future<void> playOrPause()`
- `Future<void> seek(Duration position)`
- `Future<void> setVolume(double volume)`
- `Future<void> disposePlayer()`

#### DASH Audio Handling
- **media_kit**: Use existing logic (`NativePlayer.setProperty('audio-files', ...)` + `setAudioTrack(AudioTrack.auto())`)
- **fvp**: Use fvp's extended API `MdkVideoPlayerPlatform.setExternalAudio(playerId, audioUrl)`. Requires accessing the platform interface. If not feasible during this refactor, fall back to video-only for fvp and document as a known limitation.

#### Heartbeat
- Move `_startHeartbeat` logic from `video_detail_page.dart` into `MediaS`.
- Store `bvid` and `cid` as internal state.
- Use `Timer.periodic(Duration(seconds: 15), ...)` calling `VideoHttp.heartBeat`.
- Cancel timer on `disposePlayer()` or when new video loads.

#### Audio Service Integration
- `BaseAudioHandler.play()` / `pause()` should delegate to the active player's `play()` / `pause()`.

#### Widget Access
- `Widget buildVideoWidget()` — returns either `MediaKitVideoV(controller: _mkController!)` or `FvpVideoV(controller: _fvpController!)`. Alternatively, expose the raw controller and let `VideoPlayerV` decide.

### 5. `lib/feature/video/video_detail_page.dart` (Major refactor)
Remove all player-related logic.

#### State to Remove
- `Player _player`
- `VideoController _controller`
- `Timer? _heartbeatTimer`
- `_loadMedia()` method
- `_startHeartbeat()` method

#### State to Keep
- `_currentCid`
- `_relatedVideos` / `_loadingRelated`
- `_loadDetail()`, `_loadRelatedVideos()`
- `_switchPage()`, `_showQualityPicker()`
- All layout widgets (`_buildOwnerRow`, `_buildStatRow`, `_buildVideoInfo`, etc.)

#### Player Widget Replacement
Replace `_buildPlayer()` with:
```dart
Widget _buildPlayer(PlayUrlModel? playUrl) {
  return AspectRatio(
    aspectRatio: 16 / 9,
    child: VideoPlayerV(
      playUrl: playUrl,
      onQualityTap: playUrl != null ? () => _showQualityPicker(playUrl) : null,
      onFullscreen: _pushFullscreen,
    ),
  );
}
```

#### Lifecycle Changes
- `initState()`: Call `WidgetsBinding.instance.addPostFrameCallback((_) => _loadDetail())` only. No player initialization.
- `dispose()`: Call `MediaS.i.disposePlayer()` instead of `_player.dispose()`.

#### Page Switching
- `_switchPage()` should call `MediaS.i.initAndLoad(service.playUrl)` after `service.selectPage(index)` updates the play URL.

#### Quality Switching
- After `service.loadPlayUrl()` returns, call `MediaS.i.initAndLoad(service.playUrl)`.

#### Fullscreen
- `_FullscreenPlayer` should be updated to use `VideoPlayerV` instead of directly taking `controller` and `player`. Or simply use a full-screen `VideoPlayerV` without additional controls.

### 6. `pubspec.yaml`
Add `video_player: ^2.10.1` as an explicit dependency (currently only transitive via fvp).

## Implementation Order

1. **Add `video_player` to `pubspec.yaml`** and run `flutter pub get`.
2. **Create `media_kit_video_v.dart`** — simple wrapper, no logic.
3. **Create `fvp_video_v.dart`** — simple wrapper, no logic.
4. **Refactor `media_s.dart`** — implement the full service with both backends, unified streams, heartbeat, and DASH audio support.
5. **Create `video_player_v.dart`** — Stack layout, dynamic backend switching, HUD that delegates to `MediaS`.
6. **Refactor `video_detail_page.dart`** — strip player logic, use `VideoPlayerV`, delegate to `MediaS`.
7. **Test & Verify** — ensure both media_kit and fvp paths compile and basic playback works.

## Open Questions / Considerations

### FVP DASH Audio
**Decision**: Video-only for fvp initially. The standard `video_player` API does not support separate audio tracks, and while FVP provides `MdkVideoPlayerPlatform.setExternalAudio()` as an extension, it requires platform-specific casting that is fragile. We will load only the video stream for fvp and add a TODO comment documenting this limitation. The media_kit backend retains full DASH audio support.

### Controller Exposure
**Decision**: `MediaS` exposes `Widget buildVideoWidget()` (Option B). The UI never touches the native controller, maximizing encapsulation. `VideoPlayerV` simply calls `MediaS.i.buildVideoWidget()` to render the video layer.

### Stream Unification
`media_kit` exposes `Stream<T>` directly. `video_player` uses `ValueListenable` via `controller`. `MediaS` should normalize both to `Stream<T>` so `VideoPlayerV` only uses `StreamBuilder`.

### Error Handling
- `MediaS` should expose an `errorStream` or `lastError` so `VideoPlayerV` can show loading/error states.
- Alternatively, keep error display in `VideoDetailPage` (as it currently does with `_buildPlayUrlError`) and only handle playback errors in `MediaS`.

### Fullscreen Mode
**Decision**: Reuse `VideoPlayerV` in fullscreen with the same HUD controls (play/pause/seek/quality). The fullscreen route will instantiate `VideoPlayerV` with the same callbacks, ensuring consistent interaction across normal and fullscreen modes. The HUD's fullscreen button will simply `Navigator.pop()` when already in fullscreen.

### Memory / Lifecycle
Since `MediaS` is a singleton, `disposePlayer()` must be called when `VideoDetailPage` disposes. If the user navigates to a new video via `pushReplacement`, the old page's `dispose()` will run, but the new page's `initState()` will immediately call `initAndLoad()`. Ensure `disposePlayer()` properly cleans up before the new player initializes.

## Risk Assessment

| Risk | Impact | Mitigation |
|------|--------|------------|
| FVP DASH audio setup doesn't work | Medium | Fall back to video-only for fvp; document limitation |
| Stream unification causes lag/jank | Low | Use `BehaviorSubject` or `StreamController` with `add` only on value changes |
| Singleton MediaS holds stale state | Medium | Thoroughly test `disposePlayer()` + `initAndLoad()` sequence |
| Breaking existing playback behavior | Medium | Preserve all existing headers, buffer config, and audio setup for media_kit |

---

## Success Criteria

- [ ] `video_detail_page.dart` contains zero direct `Player` / `VideoController` / `VideoPlayerController` references.
- [ ] All playback lifecycle (init, load, play, pause, seek, dispose, heartbeat) lives in `MediaS`.
- [ ] `VideoPlayerV` renders both media_kit and fvp players based on settings.
- [ ] HUD controls in `VideoPlayerV` dispatch to `MediaS`, not local state.
- [ ] `MediaS` dynamically selects backend using `StorageS.getSetting().playerLibrary`.
- [ ] No regression in existing media_kit DASH playback.
